# ADR-0002: Выделенный сервис ADR-Worker

**Статус:** Черновик  
**Дата:** 2026-03-20  
**Автор:** [Имя, роль/ID ACT-xxx]  
**Связанные артефакты:** UC-004, UC-005, ACT-002, ACT-006; [3-High Level Design/uc-realization.md](../3-High%20Level%20Design/uc-realization.md); ADR-0001, ADR-0003, ADR-0006  

---

## 1. Контекст
- **Инициатива/проблема:** В продуктовой модели aiExpert агент **ADR-Worker** оркестрирует подготовку и согласование ADR (UC-004) и выполняет итеративный цикл планирования, исполнения и оценки (UC-005). Размещение этой логики внутри «монолитного» API-слоя удлиняет запросы, усложняет масштабирование и обновление агента независимо от CRUD-операций.
- **Драйверы:** Изоляция длительных и LLM-зависимых шагов; независимое масштабирование воркеров; чёткий контракт с ядром системы (события/внутренний API); аудит журнала выполнения агента.
- **Ограничения:** Согласованность статусов ADR Request / ADR с ядром; требования к идемпотентности при повторах; доступ к LLM только через принятую политику (ADR-0001).
- **Допущения:** Ядро aiExpert предоставляет стабильные внутренние API или шину событий для чтения контекста и фиксации статусов; секреты LLM не хранятся в ADR-Worker, если используется LLM Access Service.

## 2. Проблема и целевой результат
- **Проблема:** Смешение синхронных пользовательских операций и многошаговой агентной обработки в одном рантайме приводит к таймаутам, сложной отладке и рискованным деплоям.
- **Цели/Success Criteria:**
  - Отдельный деплой/масштабирование сервиса ADR-Worker;
  - Документированный контракт взаимодействия с ядром (запуск сессии, обновление статусов, журнал шагов);
  - Восстановление после сбоев без дублирования побочных эффектов (идемпотентность ключевых операций).

## 3. Решение
- **Выбранный вариант:** Реализовать **ADR-Worker** как **отдельный сервис** (или отдельный процесс/деплой с явной границей), взаимодействующий с ядром aiExpert через согласованные механизмы (внутренний HTTP/gRPC, очередь сообщений и/или события).
- **Что будет сделано:**
  1. Определить контракт: триггеры от ядра (например, после UC-001), чтение контекста ADR Request / ADR, запись статусов и артефактов.
  2. Ввести журнал выполнения (audit trail) для шагов UC-005.
  3. Настроить политики ретраев, дедлайнов и остановки цикла.
  4. Связать с оркестрацией на LangGraph (ADR-0003), если граф выполняется внутри этого сервиса.
- **Границы решения:** ADR-Worker не заменяет хранение первичных данных ADR и справочников — источник истины остаётся в ядре/БД (ADR-0008). Не включает выбор фреймворка UI (ADR-0007).
- **Затронутые элементы архитектуры:** [SRV-ADR-Worker], интеграции с ядром [API internal], при необходимости [CNT-xxx]; UC-004, UC-005.

## 4. Рационализация
- **Причины выбора:** Разделение по скорости изменений и масштабу нагрузки; безопасная эволюция агента; типичный паттерн worker для фоновых процессов.
- **Компромиссы:** Дополнительная сетевая/операционная сложность; необходимость строгого контракта и версионирования сообщений.

## 5. Альтернативы
1. **Вся логика агента внутри одного FastAPI-процесса ядра** — проще деплой, хуже изоляция длительных задач и масштабирование. **Отклонено** в пользу выделенного сервиса.
2. **Serverless-only без постоянного сервиса** — подходит для редких задач; сложнее длинные сессии, состояние графа и отладка UC-005. **Отклонено** или отложено без отмены границы «отдельный рантайм агента».

## 6. Последствия

### 6.1 Технические последствия
- Появление очередей/событий или синхронных callback с ядром; необходимость корреляции `trace_id` / `session_id`.

### 6.2 Организационные последствия
- Владелец сервиса (команда платформы/ИИ); регламент изменений контракта с ядром.

### 6.3 Эксплуатация и безопасность
- Мониторинг: длина очереди, время шага, ошибки инструментов, лимит итераций UC-005; аудит журнала; доступ только из доверенной сети к внутренним API.

### 6.4 Риски
- Рассинхрон статусов между Worker и ядром. **Митигирующие меры:** транзакционные границы там, где возможно; идемпотентные ключи; периодическая сверка состояния.

## 7. План внедрения и валидации
- **План:** контракт → MVP один сценарий UC-004/005 → журнал и метрики → нагрузочный пилот → прод.
- **Проверка:** интеграционные тесты контракта; симуляция сбоев и повторов; контроль отсутствия двойных побочных эффектов.

## 8. Примечания и связанные решения
- **ADR-0001** — доступ к LLM; ADR-Worker обращается к модели через LLM Access Service.
- **ADR-0003** — LangGraph внутри/рядом с ADR-Worker.
- **ADR-0006** — ядро Python предоставляет API/события для Worker.

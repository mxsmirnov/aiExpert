# ADR-0001: Отдельный сервис доступа к LLM (внешняя или локальная)

**Статус:** Черновик  
**Дата:** 2026-03-20  
**Автор:** [Имя, роль/ID ACT-xxx]  
**Связанные артефакты:** [ADR Request, ABB, UC/SYS/SRV для потребителей LLM, COM/API интеграций, см. реестр]  

---

## 1. Контекст
Опишите ситуацию и рамки, в которых принимается решение:
- **Инициатива/проблема:** Несколько приложений и сервисов нуждаются в возможностях больших языковых моделей. Источник модели может меняться: коммерческий облачный API, self-hosted стек (например, vLLM, Ollama, TGI) или гибрид. Необходимо избежать дублирования клиентской логики, учётных данных и политик по всем потребителям.
- **Драйверы (почему сейчас):** единая точка изменения при смене провайдера; снижение риска утечек ключей; упрощение аудита и квот; возможность централизованно включать кэш, логирование запросов, safety-фильтры и версионирование промптов/моделей.
- **Ограничения:** требования к задержке и доступности; возможный запрет на прямой выход в интернет из некоторых контуров; лимиты провайдера и бюджет; необходимость поддержки on-premise без привязки кода приложений к конкретному бэкенду LLM.
- **Допущения:** потребители взаимодействуют с LLM через стабильный внутренний контракт (REST/gRPC); внешние и локальные эндпоинты совместимы с выбранным адаптером (например, OpenAI-совместимый API там, где это возможно); команда эксплуатации может выделить ресурсы на мониторинг и обновления одного сервиса.

## 2. Проблема и целевой результат
Что именно нужно улучшить/изменить и по каким критериям будет считаться успех:
- **Проблема:** Прямые вызовы LLM из множества сервисов приводят к разрозненным ключам, разным форматам ошибок и ретраев, сложности миграции между облаком и локальной моделью и к затруднённому контролю затрат и PII.
- **Цели/Success Criteria:**
  - Один согласованный API для всех внутренних потребителей;
  - Смена провайдера или URL локального инференса без изменения кода потребителей (конфигурация/фичефлаги на стороне LLM Access Service);
  - Централизованные логи запросов (с маскированием), метрики latency/ошибок/стоимости (где применимо);
  - Документированный процесс выдачи доступа и квот.

## 3. Решение
Четко зафиксируйте выбранный подход:
- **Выбранный вариант:** Ввести выделенный **LLM Access Service** (или «LLM Gateway») — единый компонент, через который все продуктовые сервисы обращаются к модели. Сервис отвечает за аутентификацию потребителей (mTLS, API key, OAuth — по стандартам организации), маршрутизацию к целевому бэкенду (облако / локальный инференс), нормализацию запросов/ответов, политики ретраев, таймаутов и ограничения скорости.
- **Что будет сделано:**
  1. Определить и задокументировать публичный контракт сервиса для потребителей.
  2. Реализовать адаптеры к целевым бэкендам (облачный API, локальный OpenAI-совместимый или иной согласованный протокол).
  3. Подключить секреты, конфигурацию маршрутов и feature flags (какой бэкенд для какого окружения/тенанта).
  4. Ввести наблюдаемость и политики безопасности (логирование, маскирование, квоты).
  5. Мигрировать потребителей с прямых вызовов на сервис поэтапно.
- **Границы решения:** Сервис предоставляет **доступ к выводу модели** и сопутствующую оркестрацию на уровне HTTP/gRPC; не заменяет полноценный ML platform для обучения моделей, RAG-хранилища и т.п., если они не входят в явную зону ответственности этого ADR. Fine-tuning и хранение датасетов могут быть вынесены в отдельные решения.

## 4. Рационализация
Краткое обоснование выбора (почему это лучше альтернатив в текущем контексте):
- **Причины выбора:**
  - Один слой абстракции над сменяющимися провайдерами и локальными инсталляциями.
  - Секреты и политики в одном месте, проще для security/audit.
  - Единые паттерны отказоустойчивости и наблюдаемости.
- **Компромиссы:** Дополнительный хоп сети и точка отказа; необходимость выделить SRE/владельца сервиса; возможное усложнение отладки без качественной корреляции запросов (trace id).

## 5. Альтернативы (рассмотренные варианты)
Перечислите основные альтернативы и почему они не были выбраны:
1. **Прямые вызовы LLM из каждого сервиса** — быстрее в короткой перспективе, но масштабируется плохо по безопасности и смене провайдера; дублирование логики. **Отклонено** в пользу централизации.
2. **Общая клиентская библиотека без отдельного runtime-сервиса** — снижает дублирование кода, но не даёт единой точки для квот, политик и смены маршрутизации без выката всех потребителей. **Отклонено** там, где нужны централизованный контроль и операционная независимость от релизов клиентов.
3. **Использование только публичного API шлюза провайдера как «единственной точки»** — не покрывает локальный инференс и смешанные сценарии; привязка к вендору. **Отклонено** для универсальности cloud + on-prem.

## 6. Последствия
Оцените влияние решения по направлениям:

### 6.1 Технические последствия
- Появление нового компонента в пути вызова; необходимость версионирования API сервиса.
- Реализация адаптеров и стратегий fallback (при наличии вторичного бэкенда).
- Возможное использование очередей или async API для длинных задач — по отдельной спецификации.

### 6.2 Организационные последствия
- Назначить владельца сервиса и процесс онбординга новых потребителей (квоты, SLA).
- Обновить гайдлайны разработки: новые интеграции с LLM только через LLM Access Service.

### 6.3 Эксплуатация и безопасность
- Мониторинг: latency, error rate, rate limit от провайдера, очередь (если есть); алерты на деградацию бэкенда.
- Аудит: кто и сколько вызывает; политика хранения логов и обработки PII.
- Управление секретами (vault/Kubernetes secrets); ротация ключей без полного редеплоя потребителей.

### 6.4 Риски
- **Риски:** Перегрузка шлюза; несовместимость форматов между бэкендами; задержки при авариях на единственном инстансе.
- **Митигирующие меры:** Горизонтальное масштабирование; чёткие контракты и интеграционные тесты на адаптеры; health checks и circuit breaker; при критичности — резервный бэкенд или read-only деградация по политике продукта.

## 7. План внедрения и валидации
- **План (этапы):**
  1. Зафиксировать API и NFR (latency, доступность).
  2. MVP: один бэкенд (облако или локальный), один тип вызова (например chat completions).
  3. Пилот с 1–2 потребителями; сравнение метрик с предыдущим прямым доступом.
  4. Расширение адаптеров, квоты, продакшен-роллаут и отключение прямых ключей у приложений.
- **Как проверяем, что решение работает:** нагрузочные и интеграционные тесты; пилотные SLO; регрессия функционала у потребителей; проверка сценария смены бэкенда на стейдже без кода клиента.

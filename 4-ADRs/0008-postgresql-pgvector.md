# ADR-0008: PostgreSQL и pgvector для данных и семантического поиска

**Статус:** Черновик  
**Дата:** 2026-03-20  
**Автор:** [Имя, роль/ID ACT-xxx]  
**Связанные артефакты:** UC-001–UC-003 (анализ и запросы по артефактам); [uc-realization.md](../3-High%20Level%20Design/uc-realization.md); ADR-0001, ADR-0006

## 1. Контекст
- **Инициатива/проблема:** aiExpert хранит ADR Request, ADR, справочники, связи с ABB и журналы. Для UC-002, UC-003 и анализа текста в UC-001 требуется **семантический поиск** и сходство документов без обязательного отдельного векторного SaaS на первом этапе.
- **Драйверы:** Одна СУБД для транзакций и векторов упрощает эксплуатацию; ACID для статусов и согласованности; зрелость PostgreSQL.
- **Ограничения:** Размер и производительность векторного поиска при очень больших объёмах могут потребовать шардирования или выделенного поискового движка позже.
- **Допущения:** Встраивания для pgvector получаются через согласованную модель (embedding API через ADR-0001 или локальный эмбеддер — уточняется политикой); миграции ведутся инструментами ядра (ADR-0006).

## 2. Проблема и целевой результат
- **Проблема:** Разделение OLTP и векторного хранилища без интеграционной стратегии усложняет транзакционные обновления индексов и консистентность.
- **Цели/Success Criteria:**
  - Транзакционное хранение доменных сущностей в **PostgreSQL**;
  - Индексы **pgvector** для поиска похожих ADR/ABB и RAG-подобных запросов;
  - Резервное копирование, мониторинг и политика миграций.

## 3. Решение
- **Выбранный вариант:** Использовать **PostgreSQL** как основную СУБД aiExpert и расширение **pgvector** для хранения эмбеддингов и векторного поиска.
- **Что будет сделано:**
  1. Схема таблиц для доменных сущностей + таблицы/колонки векторов с метаданными (модель, версия, timestamp).
  2. Стратегия обновления эмбеддингов при изменении текста ADR/ABB (синхронно/асинхронно через очередь — по NFR).
  3. Индексы (IVFFlat/HNSW — по версии PG и профилю нагрузки).
  4. Политика PII, ретеншн и шифрование на диске — по корпоративным стандартам.
- **Границы решения:** Не заменяет объектное хранилище для вложений, если они введутся. Не фиксирует выбор конкретного managed Postgres vs self-hosted — это уровень развёртывания.
- **Затронутые элементы архитектуры:** [DB-aiExpert], слой доступа в ADR-0006; UC с поиском и аналитикой текста.


## 4. Рационализация
- **Причины выбора:** Единый операционный контур; SQL + векторы; хорошая поддержка в экосистеме Python.
- **Компромиссы:** Векторный поиск в PG может уступать специализированным engine при экстремальном масштабе.

## 5. Альтернативы
1. **Отдельное векторное хранилище (Pinecone, Qdrant, OpenSearch kNN)** — масштаб и специализация; сложнее транзакционная синхронизация. **Отклонено** на первом этапе.
2. **Только полнотекстовый поиск без векторов** — дешевле, хуже для семантики UC-002/003. **Отклонено** как единственный механизм.

## 6. Последствия

### 6.1 Технические последствия
- Миграции схемы; job-ы пересчёта эмбеддингов; мониторинг размера индексов.

### 6.2 Организационные последствия
- DBA/платформенная поддержка; регламент изменений схемы.

### 6.3 Эксплуатация и безопасность
- Бэкапы, PITR при необходимости; ограничение доступа к прод-БД; аудит административных действий.

### 6.4 Риски
- Деградация запросов при росте данных. **Митигирующие меры:** профилирование, правильный выбор индекса, партиционирование, при необходимости вынесение векторов позже.

## 7. План внедрения и валидации
- **План:** схема MVP → загрузка тестовых корпусов → замеры recall/latency → прод.
- **Проверка:** качество поиска на золотом наборе запросов; нагрузочные тесты чтения.

## 8. Примечания и связанные решения
- **ADR-0001** — источник эмбеддингов, если через облако/локальную модель.
- **ADR-0006** — ORM/репозитории, миграции.
- **ADR-0003** — при хранении чекпоинтов LangGraph в БД, не конфликтовать со схемой домена (отдельные таблицы/схема `agent`).

# ADR-0004: API Gateway как единая точка входа

**Статус:** Черновик  
**Дата:** 2026-03-20  
**Автор:** [Имя, роль/ID ACT-xxx]  
**Связанные артефакты:** UC-001–UC-008 (доступ из сети); ACT-001–ACT-005; ADR-0005, ADR-0006, ADR-0007  

---

## 1. Контекст
- **Инициатива/проблема:** К aiExpert обращаются веб-клиент (Reflex), внешние системы (ACT-005) и потенциально другие клиенты. Прямое обнаружение множества сервисов клиентами усложняет TLS, аутентификацию, rate limit и наблюдаемость.
- **Драйверы:** Единая политика входа; централизованные логи и трассировка на периметре; упрощение маршрутизации к ядру и интеграционному API.
- **Ограничения:** Gateway не должен превращаться в «божественный» слой бизнес-логики; задержка и SPOF при неправильном дизайне.
- **Допущения:** Выбран конкретный продукт Gateway (Kong, Envoy, AWS API Gateway, NGINX + policy и т.д.) на этапе проектирования развёртывания — в ADR фиксируется **роль**, не обязательно вендор.

## 2. Проблема и целевой результат
- **Проблема:** Разные клиенты → разные URL/сертификаты/политики; дублирование auth на каждом сервисе.
- **Цели/Success Criteria:**
  - Один публичный периметр для согласованного класса клиентов (или логическое разделение route groups);
  - Единый формат идентификации клиента на бэкенде (claims, headers);
  - Документированная матрица маршрутов к ADR-0005 / ADR-0006.

## 3. Решение
- **Выбранный вариант:** Ввести **API Gateway** перед бэкенд-сервисами aiExpert как **единую точку входа** для внешнего и корпоративного HTTPS-трафика.
- **Что будет сделано:**
  1. Определить маршруты: UI → ядро; внешние системы → слой FastAPI интеграций (ADR-0005); административные API по политике.
  2. Включить TLS termination, аутентификацию (JWT/OAuth/API keys/mTLS по классу клиента), rate limiting, базовый WAF при необходимости.
  3. Прокидывать корреляционные идентификаторы к бэкендам.
- **Границы решения:** Не реализует доменные правила ADR/запросов — только периметр и маршрутизация. Не заменяет service mesh внутри кластера, если он вводится отдельно.
- **Затронутые элементы архитектуры:** [SYS-perimeter], маршруты к [SRV-core], [SRV-external-api]; ACT-001–005.

## 4. Рационализация
- **Причины выбора:** Стандартный паттерн для мульти-клиентских систем; централизация политик доступа.
- **Компромиссы:** Дополнительный хоп; необходимость HA и мониторинга Gateway.

## 5. Альтернативы
1. **Прямой Ingress к каждому сервису без Gateway** — меньше компонентов, больше дублирования политик. **Отклонено** для периметра aiExpert.
2. **Только BFF в FastAPI без отдельного Gateway** — возможно в малых контурах; хуже унификация для ACT-005 и внешних интеграций. **Отклонено** как основной периметр.

## 6. Последствия

### 6.1 Технические последствия
- Конфигурация маршрутов и upstream; согласование таймаутов с длинными операциями (частично решается async/очередями на ядре).

### 6.2 Организационные последствия
- Процесс заявок на новый маршрут/API ключ для внешних систем.

### 6.3 Эксплуатация и безопасность
- SLI: доступность Gateway, ошибки 4xx/5xx по маршрутам; аудит доступа; регулярное обновление политик.

### 6.4 Риски
- Перегрузка или мисоконфигурация. **Митигирующие меры:** HA, лимиты, canary конфигурации, алерты.

## 7. План внедрения и валидации
- **План:** выбор продукта/паттерна → dev/stage конфигурация → пилот с одним клиентом ACT-005 и UI → прод.
- **Проверка:** нагрузочное тестирование периметра; тесты отзыва ключей; трассировка end-to-end.

## 8. Примечания и связанные решения
- **ADR-0005** — маршрут для внешних систем.
- **ADR-0006** — upstream для внутреннего ядра.
- **ADR-0007** — браузерный клиент через Gateway к ядру.

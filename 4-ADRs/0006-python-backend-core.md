# ADR-0006: Серверное ядро aiExpert на Python

## 1. Контекст
- **Инициатива/проблема:** aiExpert требует единого серверного ядра для операций с ADR Request, ADR, справочниками, отчётностью и координацией с ADR-Worker. Стек должен быть согласован с агентами (LangGraph, Python) и с командой разработки.
- **Драйверы:** Единый язык с ADR-0002/0003; богатая экосистема для API, данных и ИИ; скорость разработки.
- **Ограничения:** GIL и async-модель там, где нужна высокая параллельность I/O; дисциплина типизации и тестов для растущего монолита/модульного монолита.
- **Допущения:** HTTP API ядра для сотрудников реализуется на **FastAPI** (родственно ADR-0005, но с другим контуром auth и маршрутами); при физически одном процессе — **чёткое модульное разделение** «внешний интеграционный» vs «внутренний UI/API».


## 2. Проблема и целевой результат
- **Проблема:** Разнородный бэкенд на нескольких языках усложняет найм, общие библиотеки и отладку цепочек с ADR-Worker.
- **Цели/Success Criteria:**
  - Документированная **слойная структура** (domain / application / adapters);
  - Покрытие критичных сценариев UC автотестами;
  - Явные границы между ядром и ADR-Worker (контракты API или события).

## 3. Решение
- **Выбранный вариант:** Реализовать **основное серверное ядро aiExpert на Python**, с **FastAPI** как основным фреймворком HTTP для внутренних потребителей (сотрудники, служебные вызовы), **делегированием** интеграционного M2M-контура согласно ADR-0005.
- **Что будет сделано:**
  1. Структурировать код: доменные модели, use cases, порты к БД (ADR-0008), клиент к ADR-Worker.
  2. Определить внутренние REST/gRPC контракты для UI (ADR-0007) и для Worker.
  3. Ввести единые практики: type hints, ruff/mypy (по политике команды), миграции БД.
- **Границы решения:** Не включает выбор Gateway (0004), оркестрацию графа (0003), хранилище векторов детально (0008 кроме использования). LLM — через ADR-0001, не напрямую из домена без политики.
- **Затронутые элементы архитектуры:** [SRV-aiExpert-core], [API-internal], все UC, затрагивающие операции через систему.

## 4. Рационализация
- **Причины выбора:** Один стек с агентным контуром; FastAPI + Pydantic + экосистема async.
- **Компромиссы:** Риск «большого приложения» без модульных границ — снимается архитектурной дисциплиной.

## 5. Альтернативы
1. **Ядро на JVM/Go, агенты отдельно на Python** — возможно при жёстких latency-требованиях; выше стоимость интеграции с LangGraph. **Отклонено** в текущем контексте.
2. **Только serverless functions без явного ядра** — плохо для сложной доменной модели ADR/справочников. **Отклонено** как основная форма ядра.

## 6. Последствия

### 6.1 Технические последствия
- Репозиторий(и), CI, миграции; возможное разделение пакетов `core` / `integrations` / `api`.

### 6.2 Организационные последствия
- Гайдлайны для разработчиков; code ownership модулей.

### 6.3 Эксплуатация и безопасность
- Health/readiness; метрики приложения; секреты через стандарт платформы.

### 6.4 Риски
- Технический долг при отсутствии границ модулей. **Митигирующие меры:** архитектурные ревью, линтеры импорта, ADR на выделение сервисов при росте.

## 7. План внедрения и валидации
- **План:** скелет слоёв → реализация ключевых UC → интеграция с Gateway и Worker → прод.
- **Проверка:** integration tests; тестовые сценарии из uc-realization.

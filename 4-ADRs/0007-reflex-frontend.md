# ADR-0007: Веб-интерфейс aiExpert на Reflex

**Статус:** Черновик  
**Дата:** 2026-03-20  
**Автор:** [Имя, роль/ID ACT-xxx]  
**Связанные артефакты:** UC-001–UC-008 (UI-сценарии для ACT-001–004); [uc-realization.md](../3-High%20Level%20Design/uc-realization.md); ADR-0004, ADR-0006  

---

## 1. Контекст
- **Инициатива/проблема:** Сотрудникам (ACT-001), архитекторам (ACT-002), администраторам (ACT-003), менеджерам (ACT-004) нужен единый веб-интерфейс aiExpert для сценариев из UC без разработки отдельного SPA/мобильных клиентов на первом этапе.
- **Драйверы:** Скорость разработки UI на Python в одной стеке с ядром (ADR-0006); единый язык для прототипов и продуктивных экранов.
- **Ограничения:** Экосистема Reflex моложе классических React-стеков; требования к UX, доступности и i18n нужно явно закладывать.
- **Допущения:** Браузерный клиент обращается к API ядра через **API Gateway** (ADR-0004); аутентификация сотрудников — по корпоративной политике (SSO/OIDC и т.д.) — детали в отдельных security ADR при необходимости.

## 2. Проблема и целевой результат
- **Проблема:** Разные технологии фронта и бэка увеличивают стоимость типизации контрактов и релизов.
- **Цели/Success Criteria:**
  - Покрыты ключевые UC для целевых акторов;
  - Устойчивые паттерны для **долгих операций** (ожидание ADR-Worker): статусы, опрос, при необходимости WebSocket;
  - Согласованный дизайн-системный минимум (компоненты, формы, таблицы).

## 3. Решение
- **Выбранный вариант:** Использовать **Reflex** для реализации веб-клиента aiExpert.
- **Что будет сделано:**
  1. Определить набор страниц/потоков под UC-001–008 для ACT-001–004.
  2. Интегрировать вызовы REST API ядра (ADR-0006), маршрутизация через Gateway (ADR-0004).
  3. Реализовать состояние загрузки/ошибок и отображение прогресса агента там, где UC предполагают ожидание.
- **Границы решения:** Не покрывает M2M API внешних систем (ADR-0005). Не определяет мобильные нативные клиенты.
- **Затронутые элементы архитектуры:** [CNT-UI / SRV-frontend], [API-internal]; UC с участием человеческих акторов.

## 4. Рационализация
- **Причины выбора:** Python full-stack; быстрый вывод экранов для внутреннего инструмента.
- **Компромиссы:** Зрелость экосистемы vs React; возможный lock-in на паттерны Reflex.

## 5. Альтернативы
1. **React/Next.js отдельной командой** — максимальная гибкость UI, другой стек и контракты. **Отклонено** на старте в пользу единого Python.
2. **Серверный рендер шаблонами (Jinja/YaTM)** — проще, хуже для интерактивных сценариев с состоянием. **Отклонено** для целевого UX.

## 6. Последствия

### 6.1 Технические последствия
- Сборка и деплой Reflex-приложения; CORS/название cookie при раздельных доменах — согласовать с Gateway.

### 6.2 Организационные последствия
- Компетенции UX/UI при необходимости; гайд по компонентам.

### 6.3 Эксплуатация и безопасность
- CSP, защита от XSS при отображении пользовательского контента из запросов; логирование действий на бэкенде (источник аудита — ядро).

### 6.4 Риски
- Ограничения Reflex для сложных виджетов. **Митигирующие меры:** оценка рискованных экранов заранее; точечные кастомные компоненты по документации Reflex.

## 7. План внедрения и валидации
- **План:** прототип ключевых потоков → интеграция с auth → UAT с архитекторами → итерации.
- **Проверка:** usability-тесты на UC-004/007; нагрузка на API со стороны UI-сценариев.

## 8. Примечания и связанные решения
- **ADR-0006** — бэкенд для UI.
- **ADR-0002/0003** — долгие процессы отображаются через статусы/уведомления из ядра.

# ADR-0003: Оркестрация агента ADR-Worker на LangGraph

**Статус:** В работе  
**Дата:** 2026-03-20  
**Автор:** [Имя, роль/ID ACT-xxx]  
**Связанные артефакты:** UC-005, ACT-006; [uc-realization.md](../3-High%20Level%20Design/uc-realization.md); ADR-0001, ADR-0002  

---

## 1. Контекст
- **Инициатива/проблема:** UC-005 описывает цикл **Планирование → Исполнение → Оценка** с ветвлениями, ожиданием внешних данных и лимитом итераций. Нужна явная модель состояний и переходов, а не разрозненные «скрипты» в коде.
- **Драйверы:** Поддерживаемость графа; персистенция чекпоинтов для длительных процессов; интеграция с инструментами (MCP, внутренние API); наблюдаемость по узлам графа.
- **Ограничения:** Зависимость от экосистемы Python/LangChain; версионирование графа при изменении бизнес-правил согласования.
- **Допущения:** Рантайм LangGraph развёрнут в составе или рядом с сервисом ADR-Worker (ADR-0002); вызовы к LLM идут через ADR-0001.

## 2. Проблема и целевой результат
- **Проблема:** Императивная оркестрация без явного графа усложняет сценарии 2a, 4a, 6a, 8a из UC-005 и аудит.
- **Цели/Success Criteria:**
  - Формальное описание узлов и рёбер (включая условные переходы);
  - Возможность сохранять/восстанавливать состояние сессии;
  - Единый журнал: какой узел выполнен, с каким результатом.

## 3. Решение
- **Выбранный вариант:** Использовать **LangGraph** как основу оркестрации для логики **ADR-Worker** (цикл UC-005 и связанные шаги UC-004).
- **Что будет сделано:**
  1. Спроектировать граф состояний (планирование, исполнение, оценка, ожидание, завершение, ошибка).
  2. Определить инструменты: вызовы MCP (ACT-006), API ядра, уведомления.
  3. Настроить персистенцию (checkpointer) и политику хранения.
  4. Задать лимиты итераций и обработку ошибок согласно UC-005.
- **Границы решения:** Не диктует выбор БД для чекпоинтов детально — согласовать с ADR-0008. Не заменяет бизнес-правила согласования — они остаются в ядре/конфигурации.
- **Затронутые элементы архитектуры:** SRV ADR-Worker, интеграции COM/API к MCP и ядру; UC-005.

## 4. Рационализация
- **Причины выбора:** LangGraph ориентирован на циклические агентные процессы с явным состоянием; экосистема совместима с Python-ядром (ADR-0006).
- **Компромиссы:** Привязка к стеку LangChain; кривая обучения; необходимость дисциплины версионирования графа.

## 5. Альтернативы
1. **Собственный конечный автомат без фреймворка** — полный контроль, выше стоимость поддержки. **Отклонено** в пользу готовой оркестрации.
2. **Temporal / другой workflow-движок** — сильнее для долгих распределённых процессов; выше операционная сложность на старте. **Отклонено** или пересматривается при масштабе.

## 6. Последствия

### 6.1 Технические последствия
- Хранилище для чекпоинтов; тесты графа (unit + интеграция с моками инструментов).

### 6.2 Организационные последствия
- Код-ревью изменений графа как изменений процессов; документация узлов для архитекторов.

### 6.3 Эксплуатация и безопасность
- Метрики по узлам и переходам; маскирование PII в логах состояния; секреты инструментов через vault.

### 6.4 Риски
- «Застревание» в цикле; несовместимость версий LangGraph/LangChain. **Митигирующие меры:** лимиты итераций (UC-005 8a); закрепление версий зависимостей; алерты на аномальную длину сессии.

## 7. План внедрения и валидации
- **План:** прототип графа на тестовых данных → персистенция → интеграция с ADR-Worker → пилот на реальном ADR Request.
- **Проверка:** прогон сценариев 2a, 4a, 6a, 8a; нагрузка на чекпоинтер.

## 8. Примечания и связанные решения
- **ADR-0002** — сервис, в котором живёт граф.
- **ADR-0001** — все вызовы LLM из узлов графа через LLM Access Service.

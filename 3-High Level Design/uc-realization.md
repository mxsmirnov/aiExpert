### UC-001: Заведение запроса на архитектурное решение

| Элемент | Описание |
| :--- | :--- |
| **ID** | UC-001 |
| **Название** | Создать ADR Request |
| **Акторы** | Сотрудник `ACT-001`<br/> Внешняя система `ACT-005` |
| **Краткое описание** | Инициатор или внещняя система регистрируют в aiExpert новый запрос |
| **Триггер** | Появление архитектурно значимого требования |
| **Предусловия** | Сотрудник или внешняя система аутентифицированы |
| **Постусловия** | |

#### Основной сценарий (Типичный ход событий)


| Шаг | Действие |
| :--- | :--- |
| 1 | Сотрудник создает проект запроса и заполняет основные поля |
| 2 | aiExpert анализирует текст запроса и предлагает связать его с некоторыми ABBs и ADRs |
| 3 | Сотрудник связывает предложенные системой ABBs и ADRs с запросом и добавляет свои |
| 4 | Сотрудник отправляет запрос на оценку |
| 5 | aiExpert анализирует текст и атрибуты запроса и определяет наиболее погруженного в тематику архитектора |
| 6 | Архитектор, назначенный на запрос, подтверждает его регистрацию в системе |
| 7 | aiExpert присваивает запросы дополнительные реквизиты и запускает агента ADR-Worker |

#### Альтернативные сценарии / Исключения
*Что может пойти не так?*

| Шаг | Условие | Действие |
| :--- | :--- | :--- |
| 4a | **Проект запроса остается в работе** | Система aiExpert сохраняет черновик запроса |
| 6a | **Архитектор возвращает проет на доработку** | Проект запроса возвращается инициатору с комментарияеми архитектора. Он исправляет запрос и отправляет на повторную оценку |
| 6b | **Архитектор закрывает запрос** | Архитектор находит типовое решением, связывает с ним запрос и закрывает его |

### UC-004: Подготовка и согласование ADR

| Элемент | Описание |
| :--- | :--- |
| **ID** | UC-004 |
| **Название** | Подготовить и согласовать ADR |
| **Акторы** | Архитектор `ACT-002` |
| **Краткое описание** | Архитектор на основании зарегистрированного ADR Request подготавливает архитектурное решение (ADR), согласовывает его с заинтересованными сторонами и фиксирует результат. Весь процесс оркестрируется агентом `ADR-Worker`. |
| **Триггер** | Для ADR Request, зарегистрированного в UC-001, требуется выработка и фиксация архитектурного решения |
| **Предусловия** | Существует согласованный ADR Request в статусе «к проработке»; назначен ответственный Архитектор; агент `ADR-Worker` активирован для данного запроса |
| **Постусловия** | Подготовленный ADR утвержден и связан с исходным ADR Request, либо зафиксировано решение об отклонении/перенаправлении запроса |

#### Основной сценарий (Типичный ход событий)

| Шаг | Действие |
| :--- | :--- |
| 1 | Агент `ADR-Worker` получает из aiExpert ADR Request, назначенный конкретному Архитектору, и создает рабочую сессию по подготовке ADR |
| 2 | `ADR-Worker` формирует для Архитектора краткое резюме запроса, связанных ABB и существующих ADR, а также предлагает структуру будущего ADR |
| 3 | Архитектор уточняет исходные данные (при необходимости просматривает связанные ABB/ADR, задает уточняющие вопросы инициатору через систему) и подтверждает готовность к подготовке решения |
| 4 | Архитектор при помощи `ADR-Worker` заполняет основные разделы ADR (контекст, драйверы, варианты решений, оценка, выбранное решение, последствия) |
| 5 | `ADR-Worker` проверяет ADR на полноту и согласованность (по шаблону и бизнес-правилам) и предлагает Архитектору доработки, если они требуются |
| 6 | Архитектор вносит правки по рекомендациям агента и переводит ADR в статус «Готов к согласованию» |
| 7 | `ADR-Worker` инициирует UC-005 «Цикл работы ИИ‑агента ADR‑Worker» для организации согласования ADR и последующей фиксации результата |
| 8 | По результатам выполнения UC-005 Архитектор при необходимости вносит окончательные правки и подтверждает финальную версию ADR |
| 9 | По завершении UC-005 `ADR-Worker` фиксирует итоговый статус ADR («Утвержден», «Отклонен», «На доработку» и т. п.) и синхронизирует его с исходным ADR Request |

#### Альтернативные сценарии / Исключения

| Шаг | Условие | Действие |
| :--- | :--- | :--- |
| 3a | **Архитектор считает исходные данные недостаточными** | Архитектор через `ADR-Worker` запрашивает у инициатора или смежных систем дополнительную информацию; ADR остается в состоянии «Требует уточнения», пока данные не будут получены |
| 5a | **`ADR-Worker` выявляет критические несоответствия или нарушения шаблона** | Агент блокирует перевод ADR в следующий статус, возвращает его Архитектору с перечнем обязательных исправлений |
| 7a | **Часть согласующих не реагирует в установленный срок** | `ADR-Worker` отправляет напоминания, эскалирует задержку менеджеру при превышении порогового срока; Архитектор может принять решение продолжить согласование с усеченным списком участников по установленным правилам |
| 8a | **Согласующие отклоняют ADR с существенными замечаниями** | `ADR-Worker` переводит ADR в статус «На доработку», возвращает Архитектору консолидированный список замечаний; после доработки процесс согласования перезапускается |
| 10a | **Принято решение не утверждать ADR** | Архитектор фиксирует через `ADR-Worker` решение об отклонении ADR, указывает причину; исходный ADR Request переводится в статус «Отклонен» или переназначается другому Архитектору/инициативе |

### UC-005: Цикл работы ИИ‑агента ADR‑Worker

| Элемент | Описание |
| :--- | :--- |
| **ID** | UC-005 |
| **Название** | Цикл работы ИИ‑агента ADR‑Worker |
| **Акторы** | MCP-источник `ACT-006` (агент `ADR-Worker`), при необходимости — Архитектор `ACT-002` и иные акторы как внешние участники цикла |
| **Краткое описание** | Агент `ADR-Worker` выполняет стандартный итеративный цикл: планирует шаги обработки ADR Request / ADR, исполняет соответствующие действия (запросы к MCP-источникам, уведомления, изменения статусов), оценивает результат и принимает решение о следующей итерации или завершении цикла. |
| **Триггер** | Запуск обработки конкретного ADR Request / ADR из UC-001, UC-004 или других сценариев, требующих автоматизированной оркестрации |
| **Предусловия** | Для агента `ADR-Worker` настроен контекст (идентификатор ADR Request / ADR, текущий статус, роли участников, бизнес-правила и политики согласования) |
| **Постусловия** | Для соответствующего ADR Request / ADR выполнен один или несколько итеративных шагов обработки; при достижении целевого состояния цикл завершен и зафиксирован итоговый статус |

#### Основной сценарий (Типичный ход событий)

| Шаг | Действие |
| :--- | :--- |
| 1 | Агент `ADR-Worker` получает актуальный контекст (ADR Request / ADR, связанные ABB и ADR, статусы согласования, список участников) из системы aiExpert |
| 2 | На этапе **Планирования** агент анализирует цели для текущей итерации (например, запуск согласования, сбор комментариев, эскалация, финализация статуса) и формирует план действий и набор шагов для выполнения |
| 3 | Агент формирует список необходимых операций: какие уведомления разослать, какие данные запросить у MCP-источников `ACT-006` / источников данных `ACT-007`, какие статусы записать, какие проверки выполнить |
| 4 | На этапе **Исполнения** агент по сформированному плану вызывает необходимые инструменты (MCP‑инструменты, API aiExpert, почтовые/уведомительные механизмы) и производит изменения в контексте ADR Request / ADR |
| 5 | Агент фиксирует все выполненные действия и их результаты в журнале выполнения для последующего аудита |
| 6 | На этапе **Оценки** `ADR-Worker` анализирует полученные результаты: изменился ли статус согласования, есть ли новые комментарии, соблюдены ли бизнес-правила и сроки, достигнута ли целевая цель итерации |
| 7 | Агент определяет, завершен ли процесс для данного ADR (например, ADR утвержден/отклонен, переведен «На доработку»), или требуется дополнительная итерация цикла |
| 8 | Если требуется новая итерация, `ADR-Worker` обновляет контекст (статусы, комментарии, дедлайны) и возвращается к шагам Планирования‑Исполнения‑Оценки для следующей итерации |
| 9 | При достижении целевого состояния агент завершает цикл, фиксирует итоговый статус ADR и ADR Request и возвращает управление в вызывающий UC (например, UC-004) |

#### Альтернативные сценарии / Исключения

| Шаг | Условие | Действие |
| :--- | :--- | :--- |
| 2a | **Недостаточно данных для планирования следующего шага** | `ADR-Worker` формирует запрос на уточнение (например, к Архитектору или инициатору) и переводит контекст в состояние ожидания ответа, приостанавливая цикл до получения данных |
| 4a | **Ошибка при вызове внешнего MCP‑инструмента или источника данных** | Агент фиксирует ошибку, повторяет операцию по политике ретраев; при повторном провале помечает шаг как неуспешный и формирует Архитектору задачу на ручное вмешательство |
| 6a | **Нарушены бизнес‑правила или политики согласования** | `ADR-Worker` прекращает текущую итерацию, откатывает часть действий (насколько возможно) и переводит ADR в статус «Проблема процесса», уведомляя ответственных лиц |
| 8a | **Достигнут лимит итераций цикла без достижения целевого состояния** | Агент завершает цикл в статусе «Требуется ручное вмешательство», уведомляет Архитектора и Менеджера, передает полный журнал итераций для анализа |

### UC-006: Получение отчёта по архитектурной работе

| Элемент | Описание |
| :--- | :--- |
| **ID** | UC-006 |
| **Название** | Получение отчёта по архитектурной работе |
| **Акторы** | Менеджер `ACT-004` |
| **Краткое описание** | Менеджер формирует запрос на отчёт по состоянию ADR Request и ADR, загрузке архитекторов и ключевым метрикам, получает агрегированный отчёт от системы aiExpert. |
| **Триггер** | Потребность в мониторинге состояния архитектурной работы (планёрка, еженедельный отчёт, инцидент, запрос руководства и т. д.) |
| **Предусловия** | В системе накоплены данные о запросах, ADR и назначениях архитекторов; у Менеджера есть права доступа к отчётам |
| **Постусловия** | Сформирован и предоставлен Менеджеру отчёт в выбранной форме (дашборд, выгрузка, PDF и т. д.) |

#### Основной сценарий (Типичный ход событий)

| Шаг | Действие |
| :--- | :--- |
| 1 | Менеджер открывает раздел отчётности системы aiExpert |
| 2 | Менеджер выбирает тип отчёта (по статусам ADR Request, по ADR, по загрузке архитекторов, по SLA и т. п.) и задаёт параметры (период, команды, теги, инициаторы) |
| 3 | Система aiExpert валидирует права доступа Менеджера к выбранным данным |
| 4 | Система формирует запрос к хранилищу данных / аналитическому слою, при необходимости привлекая ИИ‑агента для агрегации и интерпретации показателей |
| 5 | Система строит отчёт (таблицы, графики, KPI) и отображает его Менеджеру в интерфейсе |
| 6 | Менеджер при необходимости настраивает представление (фильтры, группировки, сортировки) и сохраняет преднастроенный шаблон отчёта |
| 7 | Менеджер может экспортировать отчёт (файл, ссылка, рассылка) и/или запланировать его периодическую отправку заинтересованным сторонам |

#### Альтернативные сценарии / Исключения

| Шаг | Условие | Действие |
| :--- | :--- | :--- |
| 3a | **Недостаточные права доступа Менеджера к данным** | Система уведомляет о недостаточных правах и предлагает обратиться к Администратору; отчёт не формируется |
| 4a | **В исходных данных присутствуют ошибки или неполнота** | Система помечает проблемные участки отчёта, выводит предупреждения и рекомендации (например, «часть ADR без статуса», «несогласованные запросы старше N дней») |
| 7a | **Не удалось выполнить экспорт или настройку расписания** | Система сообщает об ошибке и предлагает повторить операцию или выбрать другой способ доставки отчёта |

### UC-007: Управление справочниками системы

| Элемент | Описание |
| :--- | :--- |
| **ID** | UC-007 |
| **Название** | Управление справочниками системы |
| **Акторы** | Администратор `ACT-003` |
| **Краткое описание** | Администратор просматривает и редактирует мастер‑справочники aiExpert (типы запросов, статусы, шаблоны ADR, категории ABB, роли и др.), обеспечивая корректность и актуальность конфигурации системы. |
| **Триггер** | Появление новых требований к процессу, изменении организационной структуры или политики, необходимость корректировки справочников |
| **Предусловия** | У Администратора есть права на изменение справочников; определены правила и регламенты изменения справочных данных |
| **Постусловия** | Справочники обновлены и применяются ко всем последующим операциям в системе; изменения протоколируются |

#### Основной сценарий (Типичный ход событий)

| Шаг | Действие |
| :--- | :--- |
| 1 | Администратор заходит в раздел управления справочниками aiExpert |
| 2 | Администратор выбирает нужный справочник (типы ADR Request, статусы ADR, шаблоны ADR, классификатор ABB и т. д.) |
| 3 | Система отображает текущий состав записей справочника с возможностью фильтрации и поиска |
| 4 | Администратор добавляет, изменяет или помечает к архивированию записи справочника, указывает необходимые атрибуты (коды, наименования, описания, признаки активности) |
| 5 | Система проверяет изменения на целостность и отсутствие конфликтов (например, использование записей в активных ADR Request / ADR) |
| 6 | При успешной проверке система применяет изменения и фиксирует их в журнале аудита, обновляя конфигурацию для всех новых операций |

#### Альтернативные сценарии / Исключения

| Шаг | Условие | Действие |
| :--- | :--- | :--- |
| 5a | **Обнаружены конфликты или нарушения целостности** | Система блокирует применение изменений, показывает Администратору список конфликтов (например, «статус используется в X активных запросах») и предлагает скорректировать правки |
| 4a | **Администратор пытается удалить критическую запись** | Система предлагает вместо удаления пометить запись как неактивную и предупреждает о возможных последствиях |

### UC-008: Управление архитектурной командой

| Элемент | Описание |
| :--- | :--- |
| **ID** | UC-008 |
| **Название** | Управление архитектурной командой |
| **Акторы** | Администратор `ACT-003` |
| **Краткое описание** | Администратор добавляет новых Архитекторов, актуализирует их профили (компетенции, домены, роли, доступы), что позволяет системе aiExpert и агенту `ADR-Worker` корректно назначать запросы. |
| **Триггер** | Выход/приход сотрудников, изменение ролей и зон ответственности, реструктуризация команд |
| **Предусловия** | У Администратора есть доступ к кадровым данным и права на управление пользователями / ролями в aiExpert |
| **Постусловия** | Список Архитекторов и их атрибуты обновлены; информация используется механизмами назначения запросов и отчётности |

#### Основной сценарий (Типичный ход событий)

| Шаг | Действие |
| :--- | :--- |
| 1 | Администратор открывает раздел управления пользователями / архитектурной командой |
| 2 | Администратор создаёт нового пользователя‑Архитектора или выбирает существующего для редактирования |
| 3 | Администратор заполняет/обновляет профиль: ФИО, роль, компетенции, домены ответственности, принадлежность к командам, уровень seniority, рабочую нагрузку по умолчанию |
| 4 | Система проверяет корректность данных и отсутствие дубликатов, при необходимости предлагает объединить записи |
| 5 | Администратор назначает Архитектору необходимые права доступа в системе aiExpert |
| 6 | Система сохраняет изменения, обновляет справочник архитекторов и уведомляет заинтересованные системы/агенты (например, `ADR-Worker`) о новых или изменённых профилях |

#### Альтернативные сценарии / Исключения

| Шаг | Условие | Действие |
| :--- | :--- | :--- |
| 4a | **Обнаружен возможный дубль пользователя** | Система помечает потенциальный дубль, предлагает Администратору сравнить записи и принять решение об объединении/отклонении |
| 5a | **Конфликт ролей или прав доступа** | Система сигнализирует о конфликте (например, пересечение с административными ролями), предлагает согласование с ответственными за безопасность или применение безопасного набора прав |

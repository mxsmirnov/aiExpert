## Правила идентификации архитектурных объектов

В этом документе описано:

- **каким типам объектов присваиваются идентификаторы;**
- **какие префиксы используются;**
- **в каком файле находится master‑справочник (полный список) для каждого типа.**

Все идентификаторы имеют вид `<ПРЕФИКС>-NNN`, где `NNN` — трёхзначный номер с ведущими нулями (001, 002, ...).

---

## 2. Общие правила

- **Один тип — один master‑справочник.**  
  Объект любого типа впервые появляется и описывается только в своём master‑файле.

- **Во всех остальных документах используются только ссылки по идентификатору.**  
  Например: `UC-001`, `ACT-003`, `ENT-002`, `CNT-001`, `SYS-005`, `API-004` и т.д.

- **Идентификаторы не переиспользуются.**  
  Если объект удалён или признан неактуальным, его ID не назначается другим объектам.

## 3. Правила по типам объектов

### 3.1. UC — варианты использования (Use Case)

- **Префикс:** `UC`
- **Master‑справочник:** `3-High_Level_Solution_Design/uc-realization.md`
- **Содержимое master‑справочника:**
  - Таблица/список с полями: `ID (UC-xxx)`, `Название`, `Основной актор (ACT-xxx)`, `Краткое описание`.

**Как добавить новый UC:**

1. Определить новый сценарий (цель пользователя/системы).
2. Внести строку в `uc-realization.md` с новым `UC-0NN`.
3. В других документах (logical view, interaction view, диаграммы) ссылаться только по `UC-0NN`.

---

### 3.2. ACT — акторы (Actors)

- **Префикс:** `ACT`
- **Master‑справочник:** `3-High_Level_Solution_Design/uc-realization.md`
- **Кого обозначает:** внешние по отношению к системе роли пользователей и внешние организации/подсистемы, выступающие именно как участники сценария.

**Как добавить новый ACT:**

1. В master‑списке акторов в `uc-realization.md` завести запись с новым `ACT-0NN` (имя, описание роли).
2. Для акторов-систем добавь в описание признак system
3. В сценариях UC и других документах использовать только `ACT-0NN` + человекочитаемое имя.

---

### 3.3. ENT — сущности предметной области (Entities)

- **Префикс:** `ENT`
- **Master‑справочник:** `3-High_Level_Solution_Design/logical-view.md`
- **Кого обозначает:** устойчивые доменные объекты, которые хранятся и имеют состояние (таблицы БД, агрегаты, документные сущности и т.п.).

**Как добавить новый ENT:**

1. В разделе «Сущности (Entities)» файла `logical-view.md` добавить запись с новым `ENT-0NN`.
2. В остальных документах (UC, interaction, API‑описания) ссылаться на сущность по `ENT-0NN`.

---

### 3.4. CNT — контейнеры (Containers)

- **Префикс:** `CNT`
- **Master‑справочник:** `3-High_Level_Solution_Design/deployment-view.md`
- **Кого обозначает:** логические/технические контейнеры развертывания:
  - приложения, сервисы, базы данных, очереди и т.д. на уровне C4 Container / deployment view.

**Как добавить новый CNT:**

1. В `deployment-view.md` в разделе контейнеров добавить запись с `CNT-0NN`.
2. В диаграммах развертывания и описаниях указывать контейнеры по `CNT-0NN`.

---

### 3.5. COM — компоненты (Components)

- **Префикс:** `COM`
- **Master‑справочник:** `3-High_Level_Solution_Design/deployment-view.md`
- **Кого обозначает:** логические компоненты внутри контейнеров:
  - модули, сервисы в составе приложения, отдельные процессоры, воркеры, adapter‑слои и т.п.

**Как добавить новый COM:**

1. В `deployment-view.md` в разделе компонентов добавить запись с `COM-0NN` и связью с соответствующим `CNT-xxx`.
2. В текстовых описаниях и диаграммах ссылаться на компоненты по `COM-0NN`.

---

### 3.6. INT / API — интеграции и API (синонимы)

- **Префикс:** `API` 
- **Master‑справочник:** `3-High_Level_Solution_Design/interaction-view.md`
- **Кого обозначает:**
  - логические каналы взаимодействия между CNT/COM/SYS;
  - прикладные API (REST/gRPC/GraphQL и т.п.), если они описываются на уровне контракта.

**Как добавить новый API:**

1. В таблице взаимодействий файла `interaction-view.md` создать строку с новым `INT-0NN` или `API-0NN`.
2. Указать:
   - `Откуда` / `Куда` (через `CNT-xxx`, `COM-xxx`, `SYS-xxx` или `BND-xxx`, если это UI‑канал);
   - `Что передаётся / цель` (при необходимости со ссылками на `ENT-xxx`);
   - технологию, формат данных, синхронность и т.п.
3. В других документах ссылаться на интеграцию/API по её идентификатору (`INT-0NN` / `API-0NN`).

---


### 3.7. SYS — внешние системы / сервисы (Systems)

- **Префикс:** `SYS`
- **Master‑справочник:** `5-Enterprise_Landscape/services.md`
- **Кого обозначает:** внешние по отношению к описываемому решению корпоративные системы и сервисы (CRM, DWH, платёжные шлюзы и т.п.).

**Как добавить новый SYS:**

1. В `services.md` добавить строку с новым `SYS-0NN`.
2. Во всех других документах (context, interaction, deployment) ссылаться на внешние системы по `SYS-0NN`.

---

### 3.8. SRV — внешний интрефейс

- **Префикс:** `SRV`
- **Master‑справочник:** `5-Enterprise_Landscape/services.md`
- **Кого обозначает:** интерфейсы внешних по отношению к описываемому решению корпоративных системы (CRM, DWH, платёжные шлюзы и т.п.).

**Как добавить новый SRV:**

1. В `services.md` добавить строку с новым `SRV-0NN`.
2. Во всех других документах (context, interaction, deployment) ссылаться на внешние системы по `SYS-0NN`.

---

## 4. Как найти объект по идентификатору

- **UC, ACT:** открыть `3-High_Level_Solution_Design/uc-realization.md` и найти запись по `UC-xxx` или `ACT-xxx`.
- **ENT:** открыть `3-High_Level_Solution_Design/logical-view.md`.
- **CNT, COM:** открыть `3-High_Level_Solution_Design/deployment-view.md`.
- **SYS, SRV:** открыть `5-Enterprise_Landscape/services.md`.
- **API:** открыть `3-High_Level_Solution_Design/interaction-view.md`.

При добавлении нового объекта сначала обновляется его master‑справочник, затем идентификатор используется во всех остальных документах.
